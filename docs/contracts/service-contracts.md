# Service Contracts Overview

## 1. Purpose
This document defines the strict boundaries and communication rules between microservices in the E-Commerce Platform. Breaking these contracts requires a major version bump in the API or Event schema.

## 2. API Contracts
All REST APIs must adhere to the OpenAPI 3.1 specifications located in `docs/api/openapi/`. Services must treat these specifications as rigid contracts.

## 3. Event Contracts
All async events published to RabbitMQ must adhere to the JSON schemas defined in the Event Catalog. Consumers must implement the Tolerant Reader pattern, ignoring any unrecognized JSON fields added in minor updates.

## 4. Internal Headers
The API Gateway strips all external authentication headers and injects trusted internal headers before forwarding requests to backend services:
*   `X-User-Id`: The UUID of the authenticated user.
*   `X-User-Role`: The role of the authenticated user (e.g., `CUSTOMER`, `ADMIN`).
*   `X-Correlation-Id`: A unique trace ID generated by the gateway for distributed tracing.

Backend services MUST NOT trust `X-User-Id` if the request originates from outside the cluster. Network policies ensure backend services only accept traffic from the API Gateway.
